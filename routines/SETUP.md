# Generating the feed with a Claude Routine

The Wire's briefings are generated by a **Claude Code Routine** instead of the
Worker calling the Anthropic API on a cron. A routine runs as a full Claude Code
session on Anthropic's cloud and is **billed against your Claude subscription,
not metered API credits** — and its web search is included, so there are no
per-search fees. That's the whole point: it moves the recurring research off the
API meter.

> **What the routine owns, and what it doesn't.** The routine owns the **shared
> feed** (its own schedule, 3×/day) **and every personalised build**: the prompt
> file branches on a `PERSONALISED BUILD REQUEST` marker in the run input, the
> Worker fires it per user via the API trigger below, and `/api/ingest` routes a
> `userId` payload into that user's feed. Routines run asynchronously with a
> daily run cap (Pro 5 / Max 15), so the things that must answer a user *waiting
> on the page* — onboarding pitches and desk previews — still use the live
> (metered) Anthropic API, as does the legacy fallback when `FEED_SOURCE` is
> unset.

Docs: <https://code.claude.com/docs/en/routines>

## How it fits together

```
Claude Routine (subscription)            Cloudflare Worker
─────────────────────────────            ─────────────────
 reads routines/briefing-prompt.md
 GET /api/recent  ────────────────────▶  returns recently-served headlines
   x-ingest-key: <secret>                 (so the routine skips repeats)
 web-searches the desks
 writes items + the podcast script
 assembles one JSON payload
 POST /api/ingest  ───────────────────▶  validates, dedups (seen record +
   x-ingest-key: <secret>                 recency gate), fetches story images,
   optional "userId": routes the          merges into today's running feed in
   payload to that user's feed            KV, then pre-renders the podcast
                                          (awaited) so play is instant.
                                          Served at /api/today.
```

The Worker endpoint is dormant until `INGEST_SECRET` is set, so nothing changes
until you finish the steps below.

## One-time setup

### 1. Set the shared secret on the Worker

Pick a long random secret (e.g. `openssl rand -hex 24`). Add it as an encrypted
Worker secret — **don't** put it in `wrangler.toml`:

```
wrangler secret put INGEST_SECRET
```

(or Cloudflare dashboard → Worker → Settings → Variables → add **INGEST_SECRET**,
encrypted).

### 2. Tell the Worker the routine now owns the shared feed

So the cron stops generating (and overwriting) the shared feed with a metered
API build, set a plain var in `wrangler.toml` under `[vars]`:

```toml
FEED_SOURCE = "routine"
```

Leave it unset to keep the old API cron behaviour (useful as a fallback). This
switch also moves **personalised builds** onto the routine (fired via the API
trigger below); only the onboarding pitch refresh and desk previews stay on the
metered API either way.

### 3. Create the routine

At <https://claude.ai/code/routines> → **New routine** (or the Desktop app →
Routines → New → **Remote**):

- **Prompt** — paste exactly:

  > Read `routines/briefing-prompt.md` from the cloned repository and follow it
  > exactly. Use the `INGEST_URL` and `INGEST_SECRET` environment variables.

- **Repository** — `Julian-Elliott/the-wire` (so it can read the prompt file).
- **Model** — your choice; a capable model gives better research.
- **Environment** — this is the step that's easy to miss:
  - **Network access → Custom**, and add `desk.databased.business` to **Allowed
    domains** (keep "include default list" ticked). Without this the POST fails
    with `403 host_not_allowed` — the default allowlist blocks your own domain.
  - **Environment variables** — add:
    - `INGEST_URL` = `https://desk.databased.business/api/ingest`
    - `INGEST_SECRET` = the same secret you set in step 1.
- **Connectors** — remove all of them; this routine doesn't need any.
- **Trigger** — **Schedule**. The web form's smallest preset is hourly; to run
  3×/day, create it then run `/schedule update` in the CLI and set the cron
  `0 6,12,18 * * *` (matches the old Worker cron; minimum interval is 1 hour).

### 4. Test it

On the routine's detail page click **Run now**, open the run, and confirm it
finishes with `ok: true` and `accepted` > 0. Then load
<https://desk.databased.business> — the shared feed should show the routine's
stories. Remember: a green run status only means the session didn't crash; open
the run to confirm the POST actually returned 200.

## Refresh trigger (on-demand fires)

The schedule alone only gets you the three daily shared editions. Everything
on-demand — the in-app **Refresh** button *and all personalised builds* (desk
edits, the cron's per-user rebuilds) — works by the Worker **firing the
routine's API trigger**. Without this step, routine-mode Refresh and
personalised feeds silently do nothing.

### 1. Create the API trigger

On the routine's Edit page (claude.ai/code/routines) → **Add another trigger**
→ **API**. This gives you a fire URL like
`https://api.anthropic.com/v1/claude_code/routines/<routine_id>/fire` and a
bearer token.

### 2. Give both to the Worker — as encrypted secrets

```
wrangler secret put ROUTINE_FIRE_TOKEN
wrangler secret put ROUTINE_FIRE_URL
```

**Why secrets, even for the URL:** `wrangler deploy` replaces the Worker's
plain vars with whatever is in `wrangler.toml` `[vars]` — a value set only in
the dashboard as a plain var is **wiped on every push-to-main deploy**, leaving
a dead Refresh button. Encrypted secrets survive deploys. (Committing
`ROUTINE_FIRE_URL` uncommented in `[vars]` also works — it isn't sensitive on
its own — but the secret route keeps both values in one place.)

### 3. Know the throttles and the run budget

Fires are deliberately rationed to protect the routine's daily run cap
(Pro 5 / Max 15 runs per day):

- **Per-feed throttle** — at most one fire per `ROUTINE_FIRE_MIN_SECONDS`
  (default **900s**) per target (the shared feed, and each user separately),
  plus a global window across all personalised fires. A throttled Refresh
  returns `throttled` with a `retryInSec` rather than firing.
- **Daily cap on on-demand fires** — `ROUTINE_FIRE_DAILY_MAX` (default **15**)
  per London day, counted only when a fire actually starts.
- **The Worker cron bypasses that daily cap** (`noDailyCap` — it's the
  controlled scheduler). So the real daily budget is:
  **3 scheduled shared runs + up to `MAX_PERSONALISED_PER_CRON` (default 8)
  personalised fires × 3 cron runs + on-demand fires.** With several active
  personalised users that can exceed a Max plan's 15-run cap —
  `MAX_PERSONALISED_PER_CRON` (and `ROUTINE_FIRE_DAILY_MAX`) are the levers to
  keep the arithmetic inside your plan.

## Rotating the secret

Set a new value with `wrangler secret put INGEST_SECRET`, update the routine's
`INGEST_SECRET` env var to match, and run it again.
